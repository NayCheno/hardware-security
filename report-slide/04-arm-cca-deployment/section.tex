\begin{frame}[t,shrink=6]{Shelter: CCA 用户态隔离：解决方案}
\DeckKicker{解决方案}
\SlideMeta{zhang2023shelter}{E1 / Foundational}{local\_pdf\_verified}
{\large\bfseries\color{Ink} Shelter 的设计关键是 multi-GPT：用多个 GPT 视图控制同一物理内存在不同上下文中的可见性。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}
    \fcolorbox{Rule}{Paper}{%
  \begin{minipage}[t]{0.97\linewidth}
    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item Monitor 在 Root world 管理 Shelter 生命周期，并协调 Normal world OS、SPM 和 RMM 的边界。
  \item multi-GPT 让 Shelter memory 在不同执行阶段拥有不同访问属性。
  \item shadow GPT construction 降低 GPT 构造成本。
  \item 这种做法把 CCA 的 granule 权限从 VM 级抽象转成应用级隔离工具。
\end{itemize}
    \end{minipage}
    \vspace{1mm}
  \end{minipage}}
  \end{column}

      \scriptsize \begin{itemize}
  \item 01. 分配 Shelter memory
  \item 02. 构造 Shelter GPT
  \item 03. 切换访问权限
  \item 04. 进入隔离执行
  \item 05. 退出并清理 memory/cache

{\large\bfseries\color{Ink} 容器的问题不是性能，而是弱隔离；虚拟机能补安全但会破坏容器轻量化目标。\par}\vspace{1.8mm}

  \item 把容器放进完整 VM 能增强隔离，却引入 heavyweight virtualization 和更高运行成本。
  \item RContainer 试图用 CCA primitive 让容器保持轻量，同时从 untrusted OS 中隔离。

\begin{frame}[t,shrink=6]{NanoZone: CCA 细粒度内存保护：研究背景}
\DeckKicker{研究背景}
\SlideMeta{liu2025nanozone}{E3 / Draft-not-ratified}{local\_pdf\_verified}
{\large\bfseries\color{Ink} CVM 级隔离不能自动解决 CVM 内部的大 TCB 问题，特别是内核和大型应用中的 privilege-escalation bug。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}
    \fcolorbox{Rule}{Paper}{%
  \begin{minipage}[t]{0.97\linewidth}

      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item 现代 Linux 和安全关键应用代码量很大，CVM 内部漏洞仍可能泄露 secrets。
  \item Realm/CVM 保护的是对外部 host 的隔离，不等于内部最小权限。
  \item 细粒度内存保护希望把 session key、crypto state、sensitive module 与其它代码分开。
  \item NanoZone 因此把问题从 inter-VM isolation 转到 intra-CVM compartment。
