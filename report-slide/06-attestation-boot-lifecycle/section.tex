\begin{frame}[t,shrink=6]{SWATT: 软件 attestation 早期代表：内容摘要}
\DeckKicker{内容摘要}
\SlideMeta{seshadri2004swatt}{E1 / Foundational}{local\_pdf\_verified}
{\large\bfseries\color{Ink} SWATT 的核心贡献是不用专用安全硬件，也尝试让 verifier 判断 embedded device 当前内存是否被篡改。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}

    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item 论文面向 embedded devices，假设攻击者可能修改软件或配置。
  \item verifier 发送 challenge，device 运行精心设计的 checksum procedure。
  \item 安全直觉是攻击者若要隐藏改动，就需要额外模拟或搬运内存，从而导致时间差。
  \item 它是远程软件 attestation 的历史起点，但 threat model 很强依赖时序和硬件假设。
\end{itemize}

  \item 02. device 执行 checksum procedure
  \item 03. 伪随机遍历 memory
  \item 04. 返回 checksum 和 timing

\begin{frame}[t,shrink=6]{SWATT: 软件 attestation 早期代表：解决方案}
\DeckKicker{解决方案}
\SlideMeta{seshadri2004swatt}{E1 / Foundational}{local\_pdf\_verified}
{\large\bfseries\color{Ink} SWATT 的方案设计围绕一个目标：让恶意设备既返回正确 checksum 又不超时变得困难。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}
    \fcolorbox{Rule}{Paper}{%
  \begin{minipage}[t]{0.97\linewidth}
    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item checksum procedure 使用伪随机 memory traversal，减少攻击者提前准备的空间。
  \item 实现强调 tight loop 和硬件特定时序，让额外分支或模拟内存访问可被 verifier 观察。
  \item RC4 PRG 用于生成 memory access sequence。
  \item 这种机制适合讲 RA 思想起源，但今天不能直接套到复杂 TEE 或云服务器。
\end{itemize}
    \end{minipage}
    \vspace{1mm}
  \end{minipage}}
  \end{column}
  \begin{column}{0.49\textwidth}

      \PanelTitle{checksum 约束逻辑}
      \scriptsize \begin{itemize}
  \item 01. nonce 初始化 PRG
  \item 02. 生成伪随机地址
  \item 03. 读取 memory word
  \item 04. 更新 checksum state
  \item 05. 返回结果和时间
\end{itemize}

\SlideMeta{rats\_rfc}{E0 / Spec-standard SOTA}{local\_pdf\_verified}
{\large\bfseries\color{Ink} RATS 的方案价值是把证据、背书、策略和结果拆开，让 attestation 可以组合进 boot、lifecycle 和 channel binding。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}

      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item Evidence 是 attester 对自身状态的声明，不等于最终授权。
  \item Endorsement 给 verifier 提供制造商或平台根的参考信息。
  \item Verifier 根据 appraisal policy 产出 attestation result。
  \item Relying party 再把结果用于 key release、connection admission 或 workload placement。
\end{itemize}

  \item 01. collect measurements
  \item 02. produce evidence
  \item 03. fetch endorsements
  \item 04. appraise evidence
  \item 05. emit result

\begin{frame}[t,shrink=6]{VRASED verified RA co-design：研究背景}
\DeckKicker{研究背景}
\SlideMeta{nunes2019vrased}{E1 / Peer-reviewed SOTA}{local\_pdf\_verified}
{\large\bfseries\color{Ink} VRASED 的背景问题是 RA 不能只靠一段软件，key secrecy、atomicity 和 control-flow 都必须被硬件约束。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]

    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}

      \scriptsize \begin{itemize}
  \item SWATT 类方法依赖 timing 和软件约束，难以覆盖更强攻击者。
  \item TPM 等方案可提供硬件根，但可能不适合所有低端设备或形式化证明目标。
  \item RA 安全性需要保证 attestation key 不泄露、证明代码不可被中断或篡改、测量范围明确。
  \item VRASED 因此把最小硬件模块和 verified software routine 组合起来。
\end{itemize}

\midrule

\begin{frame}[t,shrink=6]{VRASED verified RA co-design：解决方案}
\DeckKicker{解决方案}
\SlideMeta{nunes2019vrased}{E1 / Peer-reviewed SOTA}{local\_pdf\_verified}
{\large\bfseries\color{Ink} VRASED 的设计把小硬件 TCB 放在 key 和 execution control 上，把复杂 hash/HMAC 留给可验证软件。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]

      \scriptsize \begin{itemize}
  \item HW-Mod 负责保证 Prv 的访问控制和 SW-Att 的安全执行。
  \item SW-Att 负责读取 memory 并调用 verified HMAC-SHA256。
  \item 硬件和软件分别验证，再组合成端到端 RA security argument。
  \item 这种拆分说明 RA 的 TCB 不必很大，但必须覆盖正确位置。
