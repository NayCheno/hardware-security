\section{Attestation、boot、lifecycle}

\DirectionIntro
{Attestation、boot、lifecycle}
{本方向区分 secure boot、launch attestation、runtime/control-flow evidence、evidence profile 与 verifier policy，避免把一次性启动测量写成持续可信。}
{三篇主讲材料是 \Key{seshadri2004swatt}、\Key{rats_rfc}、\Key{nunes2019vrased}；\Key{eat_rfc}、\Key{abera2016cflat}、\Key{dessouky2017lofat}、\Key{mao2025pdrima} 作为 auxiliary。}
{RFC 页只写规范语义和证据类型；verified RA 只写 simple embedded device 范围，不泛化成 CCA/CoVE 完整证据链。}

\PaperStory
{SWATT: 软件 attestation 早期代表}
{\Meta{seshadri2004swatt}{Foundational / E1 IEEE S\&P 2004}{source: local PDF verified}{claim: embedded software attestation baseline}}
{\begin{itemize}
  \item SWATT 提出软件 attestation 的早期代表方法。
  \item 它用 timing-sensitive checksum 让远程 verifier 检查设备内存内容。
\end{itemize}}
{\begin{itemize}
  \item 低端嵌入式设备缺少硬件 root of trust。
  \item 远程管理者仍需要确认设备代码完整性和启动状态。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是把软件 checksum 与严格时间假设绑定。
  \item 解决方案通过 verifier/prover challenge-response 约束篡改设备伪造完整内存 checksum 的空间。
\end{itemize}}
{\begin{itemize}
  \item Software-only remote attestation。
  \item Timing-sensitive memory checksum。
  \item Verifier/prover challenge-response model。
\end{itemize}}
{\begin{itemize}
  \item 论文在嵌入式节点场景评估。
  \item 安全性强依赖 timing、硬件和网络假设，不能直接迁移到现代复杂 TEE/SoC threat model。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}历史价值高，能解释 attestation 从软件测量走向硬件 RoT 的动因。
  \item \textbf{不足：}面对现代强对手和复杂 SoC/TEE 时适用性有限。
  \item \textbf{商业落地：}现代方案应转向硬件 RoT、标准化 evidence 和 verifier policy；SWATT 主要作为历史基线。
\end{itemize}}

\PaperStory
{RATS 架构}
{\Meta{rats\_rfc}{Spec-standard SOTA / E0 RFC 9334}{source: local PDF verified}{claim: architecture vocabulary only}}
{\begin{itemize}
  \item RATS 定义 attester、verifier、relying party、evidence、endorsements、reference values、attestation results 和 appraisal policy。
  \item 本 deck 用它统一后续 CCA、CoVE、DPU/NIC attestation 词汇。
\end{itemize}}
{\begin{itemize}
  \item 多平台 attestation 需要统一角色、消息、freshness 和 verifier policy 词汇。
  \item 没有架构词汇时，platform evidence、token encoding 和 verifier result 容易混在一起。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是把 evidence 生成、传递、评估角色与拓扑标准化。
  \item \Key{eat_rfc} 只作为辅助材料，说明实体状态和安全属性 claims 的 token 编码。
\end{itemize}}
{\begin{itemize}
  \item Attester/verifier/relying-party roles。
  \item Evidence, endorsements, reference values and appraisal policy。
  \item Attestation results and freshness vocabulary。
\end{itemize}}
{\begin{itemize}
  \item 规范/Survey，无新实验。
  \item 不能替代平台机制、token 格式实现或 CCA/CoVE 证据语义。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}跨平台 attestation 术语和 verifier policy 的权威基线。
  \item \textbf{不足：}不证明任一平台测量链充分，也不定义完整 token claim encoding。
  \item \textbf{商业落地：}标准化潜力强；风险在 evidence profile、endorsement 分发和 verifier policy 运维复杂。
\end{itemize}}

\PaperStory
{VRASED verified RA co-design}
{\Meta{nunes2019vrased}{Peer-reviewed SOTA / E1 USENIX Security 2019}{source: local PDF verified}{claim: verified RA co-design for simple embedded devices}}
{\begin{itemize}
  \item VRASED 设计并验证面向 simple embedded devices 的 hardware/software remote attestation root。
  \item 它把硬件隔离、软件 attestation routine 和形式化证明放进同一个 co-design。
\end{itemize}}
{\begin{itemize}
  \item RA 安全性依赖硬件、软件和协议共同成立。
  \item 早期方案很少对实现级安全性质做机器可检查证明，低端 IoT 设备还需要控制硬件成本。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是用少量硬件扩展保护 attestation 代码、密钥和内存访问。
  \item 软件 RA routine 与硬件约束一起被形式化验证，而不是只给协议草图。
\end{itemize}}
{\begin{itemize}
  \item Hardware/software RA root co-design。
  \item Formal verification of security sub-properties。
  \item MSP430/Basys3 FPGA implementation。
\end{itemize}}
{\begin{itemize}
  \item USENIX Security 2019 peer-reviewed paper。
  \item 论文报告 simple embedded device 原型和 verification evidence；具体资源、性能和证明规模以原文为准。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}证据强，结合形式化验证、硬件/软件 co-design 和真实原型。
  \item \textbf{不足：}目标是 simple embedded devices，不是 cloud CVM、CCA Realm 或 CoVE TVM evidence profile。
  \item \textbf{商业落地：}适合可审计 MCU/IoT RoT 和 attestation IP；风险在 proof maintenance、toolchain 和 profile 标准化。
\end{itemize}}

\DirectionSummary
{Attestation、boot、lifecycle}
{SWATT 说明早期 remote attestation 依赖软件测量和 timing 假设，现代方案需要硬件 RoT 与标准化 evidence。}
{RATS 把 attester/verifier/relying party、evidence、endorsement、reference value、appraisal policy 和 result 统一成架构词汇。}
{VRASED 说明 RA root 本身也需要硬件/软件 co-design 和机器可检查证明；C-FLAT/LO-FAT 补 runtime control-flow evidence lineage。}
{后续 CCA、CoVE、SPDM/TDISP、DPU/NIC attestation 都需要映射到 RATS/EAT 与平台特定 evidence profile。}
