\section{Accelerator / DPU / SmartNIC confidential offload}

\DirectionIntro
{Accelerator / DPU / SmartNIC confidential offload}
{本方向从 rack-level accelerator TEE 走向 cloud-scale DSA/NPU/GPU pool 与 Arm CCA accelerator workflow，并把 DPU/SmartNIC offload 纳入同一数据路径问题。}
{三篇主讲材料是 \Key{zhu2020hetee}、\Key{dhar2024cloudscale}、\Key{wang2026cage}；PORTAL、StrongBox、BlueField 只作辅助对比。}

\PaperStory
{HETEE: rack-scale heterogeneous TEE}
{\Meta{zhu2020hetee}{Foundational peer-reviewed / E1}{source: local PDF verified}{claim: rack-level accelerator TEE baseline}}
{\begin{itemize}
  \item HETEE 用 rack-level security controller 与 PCIe fabric 动态隔离 GPU/FPGA/TPU 等 accelerator。
  \item 它把 confidential computing boundary 从 CPU TEE 扩展到 rack-scale heterogeneous accelerators。
\end{itemize}}
{\begin{itemize}
  \item CPU TEE 不能天然保护 accelerator offload path。
  \item 敏感数据进入 GPU、driver、PCIe fabric 或 remote resource pool 后，host/CSP 仍可能观察或操控路径。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是把复杂 accelerator stack 留在受控 microserver/enclave 外围，用 SC 管生命周期。
  \item 解决方案让 SC 管理 assignment、加密认证、remote attestation、secure cleanup 和资源回收。
\end{itemize}}
{\begin{itemize}
  \item Rack-level Security Controller。
  \item PCIe ExpressFabric accelerator assignment。
  \item Remote attestation and secure cleanup lifecycle。
\end{itemize}}
{\begin{itemize}
  \item 真实硬件原型。
  \item 论文报告 ResNet152 inference 最大 throughput overhead 2.17\%，training 最大 overhead 0.95\%。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}foundational baseline，说明 CPU TEE 边界可以被 rack-level controller 扩展。
  \item \textbf{不足：}早于 SPDM/TDISP/CoVE-IO，不能当作 modern trusted I/O 标准或 vendor DPU attestation proof。
  \item \textbf{商业落地：}启发 DPU/SmartNIC security-controller 设计；风险在集中 TCB、设备固件可信和 production lifecycle 复杂度。
\end{itemize}}

\PaperStory
{CloudScale heterogeneous device confidential computing}
{\Meta{dhar2024cloudscale}{Peer-reviewed SOTA / E1 ACSAC 2024}{source: local PDF verified}{claim: cloud-scale DSA pool offload mechanism}}
{\begin{itemize}
  \item CloudScale 用 cloud-scale Security Controller 桥接 TEE-enabled nodes 与 legacy non-TEE DSA/NPU/GPU nodes。
  \item 它把 accelerator TEE 问题从单机设备扩展到云内 heterogeneous DSA pool。
\end{itemize}}
{\begin{itemize}
  \item 数据中心中多数 DSA 节点没有 TEE 能力。
  \item 用户要在 accelerator 性能与敏感数据保护之间取舍，DPU/SmartNIC/NPU 编排让问题更接近资源池。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是让 SC 成为 legacy non-TEE DSA 的 TEE proxy。
  \item 解决方案由 SC 执行 access control、attestation、key exchange 和 encrypted/authenticated data path。
\end{itemize}}
{\begin{itemize}
  \item Cloud-scale Security Controller。
  \item TEE / non-TEE DSA pool attestation。
  \item Encrypted and authenticated offload path。
\end{itemize}}
{\begin{itemize}
  \item 论文报告 AI、Redis、file-system workloads 平均 1.5--5\% overhead。
  \item 还报告可扩展到 2236 concurrent NPUs running CNNs。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}把 confidential heterogeneous computing 放在 cloud-scale resource pool 视角下。
  \item \textbf{不足：}SC 正确性和供应链可信是硬假设；不替代 SPDM/TDISP/PCIe IDE/IOMMU 或 vendor DPU RoT。
  \item \textbf{商业落地：}贴近云 DPU/SmartNIC/NPU resource-pool；风险在多租户编排、热插拔、failure isolation 和 production attestation。
\end{itemize}}

\PaperStory
{CAGE confidential accelerator workflow for Arm CCA}
{\Meta{wang2026cage}{Peer-reviewed SOTA / E1 TDSC 2026}{source: local PDF verified}{claim: CCA GPU/FPGA accelerator workflow}}
{\begin{itemize}
  \item CAGE 用 Arm CCA GPC/GPT 与 Monitor-side shadow task 保护 Realm 使用 GPU/FPGA accelerator 的 workflow。
  \item PORTAL 和 StrongBox 只作为 device-access 与 TrustZone-era baseline 辅助对比。
\end{itemize}}
{\begin{itemize}
  \item Arm CCA 保护 Realm memory，但 untrusted accelerator driver/runtime 会处理 code、metadata、MMIO、DMA buffer 和 completion state。
  \item 这类机制未来也会影响 DPU/SmartNIC packet-processing rules、queue descriptors 和 TLS termination keys。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是由 Monitor 从 untrusted stub 重建真实 accelerator task。
  \item 解决方案用 shadow task、Monitor verification、GPC/GPT protection 和 accelerator-specific cleanup 限制 driver/runtime。
\end{itemize}}
{\begin{itemize}
  \item Shadow task and Monitor-side verification。
  \item GPC/GPT protected accelerator buffers。
  \item GPU/FPGA workflow-specific cleanup。
\end{itemize}}
{\begin{itemize}
  \item 论文报告 GPU benchmark 开销 0.58\%--5.31\%，FPGA benchmark 开销 9.61\%--16.30\%。
  \item TCB 增量约 1301 LoC + 140 LoC；真实 production CCA hardware 和 device identity 仍需后续验证。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}当前 Arm CCA GPU/FPGA confidential accelerator 的 peer-reviewed SOTA。
  \item \textbf{不足：}Device identity、SPDM/TDISP、production CCA hardware、多租户 scheduling 和侧信道仍需额外机制。
  \item \textbf{商业落地：}可迁移到 DPU/SmartNIC offload；风险在 TCB 增量、设备身份和 scheduler 隔离。
\end{itemize}}

\DirectionSummary
{Accelerator / DPU / SmartNIC offload}
{HETEE 建立 rack-level controller 保护非 TEE accelerator 的 foundational 路线。}
{CloudScale 把问题推进到 cloud-scale DSA/NPU/GPU pool，并让 SC 类似 DPU/SmartNIC root-of-trust 候选。}
{CAGE 把 Arm CCA Realm 与 GPU/FPGA workflow 对齐；PORTAL/StrongBox 只是 device-access 与 TrustZone-era 辅助基线。}
{仍需把 vendor NIC/DPU attestation、secure vNIC/vSwitch/offload production evidence、SPDM/TDISP 与 SmartNIC local root 组合起来；BlueField OP-TEE/fTPM 只能作为 building-block evidence。}
