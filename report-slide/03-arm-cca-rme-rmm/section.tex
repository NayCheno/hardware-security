\PaperStory
{Arm Confidential Computing Architecture}
{\Meta{li2022cca}{Foundational / E1 OSDI 2022}{source: local PDF verified}{claim: mechanism and prototype claims from primary paper}}
{\begin{itemize}
  \item 提出并验证 Arm CCA 的核心设计，包括 Realm、RME、RMM、GPT/GPC 与 lifecycle。
  \item 本 deck 用它作为 Arm CCA 主线入口。
\end{itemize}}
{\begin{itemize}
  \item 云场景下 host 仍需管理资源，但不应看到 tenant workload 数据。
  \item 传统 hypervisor trust model 无法满足 confidential workload 对 host 隔离的需求。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是把资源管理和数据可见性分离。
  \item 解决方案是用硬件 granule ownership 和 RMM lifecycle 管理 Realm。
\end{itemize}}
{\begin{itemize}
  \item Realm world and host/tenant separation。
  \item Granule ownership and GPT/GPC。
  \item RMM-mediated lifecycle and attestation path。
\end{itemize}}
{\begin{itemize}
  \item OSDI 系统论文提供设计验证和原型评估。
  \item 不从二手材料复制性能值；定量细节在 slide rewrite 中只回引本地 OSDI PDF。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}把 Realm/RME/RMM 的安全边界讲清，是 Arm CCA 主线入口。
  \item \textbf{不足：}论文原型不等同于后续完整规范或量产实现。
  \item \textbf{商业落地：}潜力强；风险在硬件普及、RMM 正确性和 platform attestation 集成。
\end{itemize}}

\PaperStory
{Arm CCA 架构规范}
{\Meta{arm_cca_spec}{Spec-standard SOTA / E0}{source: source verified, PDF unavailable}{claim: public architecture semantics only}}
{\begin{itemize}
  \item 给出 CCA 架构接口、Realm、memory ownership 和 attestation 语义。
  \item 本 deck 用它固定 CCA 的规范语义。
\end{itemize}}
{\begin{itemize}
  \item 研究论文需要落到稳定的硬件/固件 ABI 和平台行为。
  \item 没有规范语义时，Realm、RME、RMM、PAS、RIPAS 和 attestation 容易被不同论文各自解释。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是用架构规范固定对象、状态、接口和安全语义。
  \item 解决方案是把 CCA/RME 相关状态和接口标准化。
\end{itemize}}
{\begin{itemize}
  \item Realm security state and PAS ownership。
  \item Memory delegation, RIPAS and lifecycle vocabulary。
  \item Attestation and platform interface semantics。
\end{itemize}}
{\begin{itemize}
  \item 规范/Survey，无新实验。
  \item Source verified 但本地 PDF 不可用：只写公开规范语义，不写不可见细节或性能 claim。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}写 CCA 机制和状态语义时最权威。
  \item \textbf{不足：}不是性能、可用性或实现正确性的证据。
  \item \textbf{商业落地：}取决于 Armv9 CCA 平台、firmware/RMM、OS 和 verifier 生态同步成熟。
\end{itemize}}

\PaperStory
{Realm Management Monitor 规范}
{\Meta{arm_rmm_spec}{Spec-standard SOTA / E0}{source: source verified, PDF unavailable}{claim: public RMM/RMI/RSI lifecycle only}}
{\begin{itemize}
  \item RMM spec 定义 Realm 管理软件接口、RMI/RSI 角色和 lifecycle。
  \item 本 deck 用它把 RMM 放在 CCA TCB 核心位置。
\end{itemize}}
{\begin{itemize}
  \item RMM 位于 CCA TCB 核心，状态机或内存管理错误会破坏 Realm 安全。
  \item RMM 既要服务 host-side 管理，又不能把 Realm 私有状态暴露给 host。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是通过规范接口约束 Realm creation、memory transition、measurement、run/exit 和 attestation 相关状态。
  \item \Key{wu2024rmm} 只作为辅助说明 RMM verification 风险，不替代规范。
\end{itemize}}
{\begin{itemize}
  \item RMI/RSI interface boundary。
  \item Realm and REC lifecycle。
  \item Granule state transition and measurement hooks。
\end{itemize}}
{\begin{itemize}
  \item 规范/Survey，无新实验。
  \item Source verified 但本地 PDF 不可用：只写公开 RMM/RMI/RSI lifecycle 概念；验证论文覆盖范围有限，不能泛化到完整 RMM。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}能把 RMM 作为 CCA TCB 核心讲清。
  \item \textbf{不足：}不证明任一 RMM 实现正确，且本地 PDF 不可用时不能写不可见细节。
  \item \textbf{商业落地：}依赖 RMM 实现验证、firmware 更新、硬件支持和 verifier 能否检查关键状态。
\end{itemize}}

\DirectionSummary
{Arm CCA / RME / RMM 基础架构}
{OSDI CCA 论文提出 Realm/RME/RMM 的基础隔离模型，把 host 管理权与 tenant 数据可见性分离。}
{Arm CCA 规范固定 Realm、PAS/RIPAS、memory lifecycle 和 attestation 等公开架构语义。}
{RMM 规范把 Realm lifecycle、RMI/RSI 和 granule state transition 变成 CCA TCB 的可审查边界。}
{仍需后续方向细化部署模型、device/accelerator I/O、interrupt ownership 和 RMM 实现验证覆盖。}
