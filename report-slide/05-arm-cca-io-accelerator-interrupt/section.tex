\begin{frame}[t,shrink=6]{ACAI: CCA accelerator execution：内容摘要}
\DeckKicker{内容摘要}
\SlideMeta{acai2023}{E3 / Draft-not-ratified}{local\_pdf\_verified}
{\large\bfseries\color{Ink} ACAI 的核心贡献是让 CCA-enabled peripheral 直接安全访问 Realm memory，而不是让 VM 先把数据复制到非安全缓冲区。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}
    \fcolorbox{Rule}{Paper}{%
  \begin{minipage}[t]{0.97\linewidth}
    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item 论文指出 confidential VM 想使用 GPU/FPGA 时，传统做法常需要 extra copy 和 software encryption。
  \item ACAI 重新设计 CCA I/O path，使 accelerator 的 bus transaction 和 address translation 也接受 granule protection checks。
  \item 它用 PCIe IDE keying、SMMU/GPT enforcement 和 realm/device lifecycle 组合保护 accelerator execution。
  \item 它是早期 draft 系统，不是 Arm CCA 标准已完成的 trusted I/O 证明。
\end{itemize}
    \end{minipage}
    \vspace{1mm}
  \end{minipage}}
  \end{column}
  \begin{column}{0.49\textwidth}
    \fcolorbox{Rule}{Panel}{%
  \begin{minipage}[t]{0.97\linewidth}
    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{encrypted-copy path vs ACAI path}
      \scriptsize \textbf{传统加密搬运}\par
Realm encrypts buffer → copy to shared memory → device computes → copy back → Realm decrypts\par
{\tiny\textcolor{Muted}{extra copies and software crypto}}\par\vspace{1.8mm}
\textbf{ACAI}\par
Realm grants device path → GPC/SMMU checks → IDE-protected PCIe path → accelerator computes\par
{\tiny\textcolor{Muted}{device transaction becomes checked path}}\par\vspace{1.8mm}

\begin{frame}[t,shrink=6]{Devlore: device interrupt protection：内容摘要}
\DeckKicker{内容摘要}
\SlideMeta{bertschi2026devlore}{E3 / Draft-not-ratified}{local\_pdf\_verified}
{\large\bfseries\color{Ink} Devlore 的核心贡献是把 device interrupt 也纳入 confidential VM 边界，阻止 host 随意伪造或重放设备中断。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}
    \fcolorbox{Rule}{Paper}{%
  \begin{minipage}[t]{0.97\linewidth}
    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item 论文指出攻击者可注入 malicious interrupts，破坏 confidential VM 的机密性和完整性。
  \item Devlore 让 CVM 注册 device interrupts，并由 monitor/RMM 检查物理中断与虚拟中断的绑定。
  \item 它关注的是 interrupt path，不是 DMA、MMIO 或 device identity 的全部问题。
  \item 它是 2026 draft，适合作为最新 interrupt protection 方向证据。
\end{itemize}
    \end{minipage}
    \vspace{1mm}
  \end{minipage}}
  \end{column}

      \scriptsize \textbf{unprotected path}\par
device event → host/GIC config → virtual interrupt → CVM handles\par

{\large\bfseries\color{Ink} 中断是 CCA I/O 边界中容易被低估的一环，因为它不搬运数据，却能改变 CVM 控制流。\par}\vspace{1.8mm}
