\PaperStory
{ACAI: CCA accelerator execution}
{\Meta{acai2023}{Draft/not ratified / E3 arXiv 2023}{source: local PDF verified}{claim: draft accelerator-execution design}}
{\begin{itemize}
  \item ACAI 研究如何用 Arm CCA 保护 accelerator execution。
  \item 它说明 CPU Realm 之外的 offload path 不能默认继承机密性。
\end{itemize}}
{\begin{itemize}
  \item Confidential workloads 需要使用 GPU/NPU/FPGA 等加速器。
  \item 如果设备 DMA、队列和驱动仍由 untrusted host 控制，Realm 内数据会在 offload 路径暴露。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是把 Realm 安全边界延伸到 accelerator execution path。
  \item 解决方案围绕 CCA-aware offload，连接设备内存访问、队列管理和 attestation。
\end{itemize}}
{\begin{itemize}
  \item CCA-aware accelerator execution path。
  \item Protected DMA and command/data movement。
  \item Driver/runtime TCB boundary for offload。
\end{itemize}}
{\begin{itemize}
  \item 规范/Survey，无新实验。
  \item 证据基础是 arXiv 系统论文原型结果；draft/not-ratified 状态下具体性能和安全结论需回引本地 PDF，不能写成标准或生产事实。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}把 Arm CCA 讨论从 CPU/VM 推到 accelerator workflow。
  \item \textbf{不足：}预印本不能代表标准化接口或量产设备能力，驱动/runtime TCB 仍是主要风险。
  \item \textbf{商业落地：}适合 AI/HPC confidential offload；风险在设备厂商支持、驱动隔离和 attestation 生态不统一。
\end{itemize}}

\PaperStory
{Devlore: device interrupt protection}
{\Meta{bertschi2026devlore}{Draft/not ratified / E3 arXiv}{source: local PDF verified}{claim: draft interrupt-protection design}}
{\begin{itemize}
  \item Devlore 关注 confidential VM 的 device interrupt protection。
  \item 它把 interrupt delivery 纳入机密计算边界，而不是只讨论 DMA。
\end{itemize}}
{\begin{itemize}
  \item 即便 DMA 受控，设备 interrupt 仍可能泄露 timing/ownership 信息。
  \item Host 也可能通过 interrupt path 干扰 confidential VM 的状态机和设备所有权。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是 interrupt 是 device ownership 的一部分。
  \item 解决方案将 interrupt delivery、device ownership 与 confidential boundary 绑定。
\end{itemize}}
{\begin{itemize}
  \item Confidential-device interrupt ownership。
  \item Interrupt delivery mediation。
  \item VM/device lifecycle coupling。
\end{itemize}}
{\begin{itemize}
  \item 规范/Survey，无新实验。
  \item 证据基础是本地已核验 arXiv PDF；本 deck 只写机制方向和证据状态，不写未复核数字或成熟部署结论。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}明确指出 interrupt 是 CCA device path 的核心安全面。
  \item \textbf{不足：}Draft evidence，落地需硬件、firmware、OS、hypervisor 和 interrupt controller 协同。
  \item \textbf{商业落地：}对云端直通设备和加速器租户隔离关键；风险在 legacy 设备兼容和中断虚拟化复杂性。
\end{itemize}}

\PaperStory
{Accelerator TEE SoK for CCA I/O}
{\Meta{sok-tee}{Background substrate / E2 NDSS 2026 SoK}{source: local PDF verified}{claim: taxonomy only}}
{\begin{itemize}
  \item Accelerator TEE SoK 为 accelerator/device TEE 提供 taxonomy。
  \item 本方向用它组织 GPU/NPU/DPU/SmartNIC 的身份、内存路径、runtime 与 attestation 问题。
\end{itemize}}
{\begin{itemize}
  \item 设备 TEE 机制比 CPU TEE 更分散。
  \item 不同 accelerator 的队列、固件、驱动和本地 root 差异很大，不能用一个 CPU TEE 模型覆盖。
\end{itemize}}
{\begin{itemize}
  \item 核心洞察是用 taxonomy 分层，而不是把所有 offload 都写成同一种 TEE。
  \item 解决方案区分设备身份、内存路径、队列、runtime、driver TCB 与 attestation。
\end{itemize}}
{\begin{itemize}
  \item Accelerator/device TEE design taxonomy。
  \item Device identity and attestation vocabulary。
  \item Memory, queue, runtime and driver TCB dimensions。
\end{itemize}}
{\begin{itemize}
  \item 规范/Survey，无新实验。
  \item Primary mechanism claim 必须回引原始论文、规范或平台文档。
\end{itemize}}
{\begin{itemize}
  \item \textbf{优点：}能把 accelerator TEE 设计空间系统化，避免只围绕 CPU Realm 叙事。
  \item \textbf{不足：}不是 Arm CCA 专用机制证据，也不是某个设备的生产部署证明。
  \item \textbf{商业落地：}有助于产品需求拆解和 vendor 对标；风险在 taxonomy 与具体硬件能力之间仍需二次验证。
\end{itemize}}
