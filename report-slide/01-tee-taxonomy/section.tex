\begin{columns}[T,totalwidth=\textwidth]

      \scriptsize \begin{itemize}
  \item 云 TEE 的承诺是让用户把敏感 workload 放到不完全可信的 server-side platform 上执行。

\begin{frame}[t,shrink=6]{Accelerator TEE 设计空间：研究背景}
\DeckKicker{研究背景}
\SlideMeta{sok-tee}{E2 / Background substrate}{local\_pdf\_verified}
{\large\bfseries\color{Ink} 只保护 CPU 内存并不能保护 accelerator workload，因为敏感数据会进入 DMA、device memory 和 accelerator runtime。\par}\vspace{1.8mm}
\begin{columns}[T,totalwidth=\textwidth]
  \begin{column}{0.45\textwidth}
    \fcolorbox{Rule}{Paper}{%
  \begin{minipage}[t]{0.97\linewidth}
    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{讲解逻辑}
      \scriptsize \begin{itemize}
  \item AI、图计算、网络和存储卸载会把秘密模型、输入数据和中间结果送到 accelerator。
  \item 传统 CPU TEE 的安全边界经常止于 CPU package 或 VM memory，不覆盖 device-local state。
  \item Accelerator TEE 的难点是 device identity、command queue、DMA buffer、memory encryption 和 attestation 要一起考虑。
  \item 这为后续 Arm CCA I/O、DPU/SmartNIC 和 confidential storage 章节提供问题边界。
\end{itemize}
    \end{minipage}
    \vspace{1mm}
  \end{minipage}}

    \vspace{1.1mm}
    \hspace{1.4mm}\begin{minipage}{0.92\linewidth}
      \PanelTitle{CPU TEE vs accelerator TEE}
      \scriptsize \textbf{CPU-centric TEE}\par
CPU memory → enclave/realm → host untrusted\par
{\tiny\textcolor{Muted}{主要保护 CPU 执行和内存}}\par\vspace{1.8mm}
\textbf{Accelerator TEE}\par
DMA buffer → device runtime → device memory → result path\par
